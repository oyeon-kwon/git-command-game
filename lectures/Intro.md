### 버전 관리 시스템의 필요성
최종.jpg 진짜 최종.jpg…. 버전관리로 인해서 고통받는 디자이너, 대학원생들의 모습

여러분은 파일을 편집 전 상태로 되돌리고 싶을 때 어떻게 하고 있나요?
가장 간단한 방법은 편집하기 전에 파일을 미리 복사해두는 것입니다. 파일과 폴더명 뒤에 편집한 날짜를 붙여주는 방식이죠. 하지만 파일을 편집할 때마다 매번 손수 복사하는 일은 번거롭기도 하고 실수할 가능성도 많습니다.

게다가 한 사람이 아니라 여러 사람이 한 파일을 가지고 일한다면? 만약 다시 되돌려야 할 경우가 생긴다면 어떤걸 되돌려야 할 지 어떻게 찾을 수 있을까요? 그래서 버전 관리를 이용해야 합니다.


**버전 컨트롤을 사용하는 이유?**
1. 파일 변경 이력을 저장할 수 있다.
2. 이전 버전을 가지고 올 수 있다. : 다시 이전 버전으로 되돌아가야 하는 상황이 발생할 경우 이전 커밋들이 기록되어 있으니 되돌아가는 것이 가능하겠죠?
3. 어떤 변경사항이 발생했는지 알아보기 쉽다. : 커밋은 이력을 남기는 작업이기 때문에 커밋을 할 때는 메시지를 꼭 작성해야 합니다. 
4. 협업하기에 좋다. : 깃헙이라는 클라우드를 이용해서 온라인 상에서 코드를 공유할 수 있기 때문에 협업에 편리합니다.
5. 백업용 : 클라우드에 소스 코드를 저장하는 방식이기 때문에 백업에도 유용합니다.

Git에서는 소스 코드가 변경된 이력을 쉽게 확인할 수 있고, 특정 시점에 저장된 버전과 비교하거나 특정 시점으로 되돌아갈 수도 있습니다.

### Git과 GitHub

버전관리를 할 수 있게 해주는 것이 git 이라는 것은 알겠는데 그렇다면 github은 무엇일까요?
두 가지를 비교해보자면
Git
소스 코드 기록을 관리하고 추적 할 수있는 버전 컨트롤 시스템입니다.
Github
Git 리포지토리를 관리 할 수있는 클라우드 기반 호스팅 서비스입니다.

즉 깃헙을 통해서 깃으로 버전 관리하는 폴더에 대해서 여러 사람들이 공유하고 접근할 수 있는 것이죠.

github: social platform for git users

유명 오픈소스 라이브러리들 공식문서의 귀퉁이에 문어 모양 아이콘이 있는 것을 보신 적이 있나요? 이게 바로 오픈 소스의 소스 코드를 깃헙에 공유한 링크인데요.
ex. 리액트 같은 경우에도 이렇게 오픈 소스로 공유가 되어 있고 여기에 contribute을 할 수 있어요. 1500여명의 contributors 가 있다는 것이 보이죠.
그리고 어떤 수정사항이 있는지 커밋 기록을 통해서 확인할 수 있고 변경된 코드도 볼 수 있습니다.

Git 설치 영상 재사용(https://urclass.codestates.com/ca66992a-d2d1-4ee0-9aed-15b7e7cded6a?playlist=401)

https://git-scm.com/book/ko/v2/%EC%8B%9C%EC%9E%91%ED%95%98%EA%B8%B0-Git-%EC%B5%9C%EC%B4%88-%EC%84%A4%EC%A0%95

Git을 설치하고 나서 가장 먼저 해야 하는 것은 사용자이름과 이메일 주소를 설정하는 것입니다. Git은 커밋할 때마다 이 정보를 사용한다. 한 번 커밋한 후에는 정보를 변경할 수 없습니다.
$ git config --global user.name "John Doe"
$ git config --global user.email johndoe@example.com
다시 말하자면 --global 옵션으로 설정하는 것은 딱 한 번만 하면 됩니다. 해당 시스템에서 해당 사용자가 사용할 때는 이 정보를 사용합니다. 만약 프로젝트마다 다른 이름과 이메일 주소를 사용하고 싶으면 --global 옵션을 빼고 명령을 실행합니다.

### Repository 에 대해 이해할 수 있다. 로컬과 원격 저장소의 차이를 이해할 수 있다.

저장소: 말 그대로 파일이나 폴더를 저장해 두는 곳
remote: 원격 온라인 상에 저장, 여러 사람이 함께 공유 가능
local: 내 컴퓨터 저장, 내 개인 전용 저장소

리액트 프로젝트는 온라인 상에 올라와있는 오픈소스입니다. 즉 원격 저장소이죠. 리액트 프로젝트에 contribute을 하기 위해서는 먼저 리액트 원격 저장소를 내 원격 저장소로 가지고 오는 작업이 필요합니다. 그 과정을 Fork 라고 합니다. 

이제 포크를 하고나면 나의 원격 저장소에 리액트 코드를 옮겨온 상태입니다. 이 코드를 수정하기 위해서는 내 컴퓨터로 가져오는 작업이 또 필요할텐데요. 그 과정을 clone 이라고 합니다. 원격 저장소에 있는 코드를 clone 명령어를 터미널창에 입력함을 통해서 내 컴퓨터로 가지고 올 수 있어요.


<img width="566" alt="스크린샷 2021-03-12 오후 6 46 26" src="https://user-images.githubusercontent.com/61301574/110922625-4191b080-8363-11eb-8e44-51b91a671096.png">

￼
내 컴퓨터에서 리액트 소스코드 변경 작업을 완료했습니다. 그렇다면 이 변경된 내용을 원격 저장소에 반대로 올려주는 작업이 필요하겠죠? 이 과정을 push 라고 합니다. 변경 내용을 커밋을 통해 커밋 메시지를 적어주고, 이 커밋을 git push 명령어를 통해서 로컬 저장소에 있던 커밋 기록을 원격 저장소로 업로드할 수 있습니다.

로컬에서 변경된 사항을 원격 저장소에 올려주기 위해서는 Push를 사용했었죠? 그런데 반대 상황도 발생할거예요. 원격 저장소에서 뭔가 변경되었을 때 로컬로 가져오는 방법은 pull 명령어를 사용할 수 있습니다.
